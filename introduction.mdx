SQLFluff is a dialect-flexible and configurable SQL linter. Designed with [ELT](https://www.techtarget.com/searchdatamanagement/definition/Extract-Load-Transform-ELT) applications in mind, SQLFluff also works with Jinja templating and dbt. SQLFluff will auto-fix most linting errors, allowing you to focus your time on what matters.

As analytics transitions from one-off work to building stable and reusable infrastructure, the same engineering principles that power software quality — consistent style, automated checks, and code review — now apply to SQL. SQLFluff fills that role for SQL teams.

## Why SQLFluff?

With larger and larger teams maintaining large bodies of SQL code, it becomes more and more important that code is not just _valid_ but also easily _comprehensible_ by other contributors. Linters enforce a consistent style and surface problems early — before they reach production.

SQLFluff brings this discipline to SQL. It is built around three components:

1. **Parser** — Lexes and parses SQL into a segment tree, handling dialect differences and whitespace precisely.
2. **Linter** — Measures parsed SQL against a configurable set of rules, reporting violations with line numbers and rule codes.
3. **Fixer** — Applies automated rewrites to resolve violations wherever possible.

SQLFluff is opinionated but flexible. It ships with sensible defaults and an extensive rule set, while allowing any team to override defaults to match their existing conventions.

## Templates supported

SQL does not lend itself well to modularity, so it is commonly templated for reuse. SQLFluff supports linting templated SQL by substituting dummy values at lint time:

- **Jinja2** (aka Jinja)
- **SQL placeholders** (e.g. SQLAlchemy parameters)
- **Python format strings**
- **dbt** (requires the dbt templater plugin)
